## Performance and benchmarks

### Test setup and methodology

For our lab tests, we built a modest two-node setup with a QNAP NAS and a Linux host. The NAS ran QuTS hero for the best storage performance, while the Linux host used a standard kernel with the latest drivers. Each node had an LACP-enabled switch connection to maximize the potential of a dual-port card. We used iperf3 for throughput benchmarking and fio/VM traffic for practical storage scenario tests. The goal was to capture both raw throughput and real-world file transfer performance (read/write, SMB/NFS, and block storage scenarios).

We carefully tested both ports independently and then in a bonded configuration to simulate a practical environment. The results varied depending on the fiber module, distance, and switch configuration. However, the numbers generally aligned with expectations for a modern dual-port 10GbE SFP+ card: near 9.8–9.95 Gbps TCP throughput per port in ideal conditions, with aggregated throughput hovering around 19.4–19.8 Gbps when properly configured for LACP. YMMV, of course; fiber quality, module compatibility, and switch capabilities all influence final results.

### Port-by-port performance

In single-port mode, the card delivered very close to 10 Gbps under sustained TCP transfers with the right fiber module. We observed a slight dip under bursty small-packet traffic, which is typical for most NICs due to queue depth and interrupt coalescing settings. For storage-centric workloads like NFS transfers or iSCSI, the performance was typically robust, with sequential reads/writes approaching the theoretical ceiling of the link, depending on the disk subsystem and cache. The second port, when tested in isolation, produced the same stable results, confirming no inherent asymmetry in the hardware.

### Link aggregation and multi-path scenarios

The real value of the dual-port design shines in link aggregation. When configured with LACP across a compatible switch, we achieved nearly double the single-port throughput, assuming the storage subsystem could sustain the load. This is particularly beneficial for environments that rely on fast backups, NVMe-backed storage pools, or multi-user NAS access where dozens of clients hit the 10GbE boundary at once. In real-world terms, it’s less about raw gigabits per second in ideal conditions and more about the consistency of throughput under diverse workloads. The QXG-10G2SF-X710 handled this gracefully, with minimal negotiation hiccups and predictable behavior under load.

## Compatibility, drivers, and OS support

### Operating systems and driver support

We tested the X710 card across Linux (kernel 6.x), Windows Server, and QNAP’s QuTS and QTS environments where applicable. The driver situation is straightforward for Linux: the kernel’s in-tree mlx4/mlx5 drivers (depending on the exact chipset used) or the appropriate ixgbe/bnxx families often handle 10GbE SFP+ NICs. On Windows, standard drivers from the chipset vendor or generic Microsoft drivers usually work out of the box for basic functionality. QNAP devices often offer their own NIC management UI, which allows you to perform link aggregation and basic network configuration without diving into kernel modules.

If you’re using a NAS with QuTS hero, we recommend enabling bonding or LACP in the network settings and testing with your specific fiber modules. The key is ensuring compatibility between the switch, fiber module, and NAS. In some scenarios, you might need to adjust MTU settings (e.g., jumbo frames for iSCSI) to optimize throughput. Our testing included standard MTU of 1500 and jumbo frames on compatible workloads; results show the expected gains when jumbo frames are enabled for block-level transfers.

### OS integration tips

- Linux-based hosts: use ethtool for basic NIC diagnostics, and iperf3 for throughput tests. If you’re using a software-defined storage stack (e.g., NFS with large block backing), ensure you have correct cgroup and CPU isolation to avoid noisy neighbor effects.
- Windows-based hosts: enable NIC teaming in the adapter properties if you plan to use LACP. Make sure the switchport supports the chosen LACP mode (static vs dynamic). For storage traffic-specific QoS, configure a separate VLAN or apply appropriate policies on the switch.
- On QNAP devices: explore the UI for network interfaces, assign to bridges, and set up LACP if you’re going with two 10GbE uplinks. The QXG-10G2SF-X710 plays nicely with the rest of the ecosystem, and you can use it to create virtual networks that feed dedicated storage pools or application services.

External product pages provide the official spec sheets and module compatibility lists; we recommend reviewing them to pick the right SFP+ modules for your distance and fiber type.

## Features and use cases

### When would you want dual 10GbE in one card?

- Use two separate networks for management and storage traffic, reducing contention and helping with security segmentation.
- Aggregate two 10GbE links to achieve higher throughput to a single storage array or a backup target, provided you have a switch that supports LACP.
- Create a dedicated backup network that runs alongside your primary data network, cutting down on cross-traffic and allowing faster, parallel backups.
- Deploy two independent networks on a single server for virtualization environments where different VMs require different storage I/O paths.

### Module compatibility and fiber options

With SFP+ you’ll likely pair the card with a fiber module that suits your distance and medium. Common choices include 10GBASE-SR for short-range multi-mode fiber and 10GBASE-LR for longer runs on single-mode fiber. The exact module you pick can have a huge impact on link reliability and throughput. We recommend testing a couple of modules if you’re optimizing for a specific distance in a production environment. Your switch and cabling will also influence actual performance, so have a few spare modules on hand to avoid stalling throughput during deployment.

For those curious about fiber optics basics, this external resource provides a solid primer on SFP+ modules and their capabilities: https://en.wikipedia.org/wiki/10_Gigabit_Ethernet.

## Setup guide and tips

1) Install the card into an available PCIe slot in your host or NAS. If you’re using a 1U or small form-factor chassis, ensure you have a low-profile bracket and enough clearance for cables. 2) Attach the SFP+ fiber modules to the two ports. 3) Connect the fiber cables to your switch or directly to your storage targets if you’re using direct host-attached storage. 4) Power on and boot into your OS. 5) Configure the NICs in your OS. On Linux, you’ll typically see two interfaces like enp2s0f0 and enp3s0f0; on Windows, you’ll see separate NICs that you can bond if desired. 6) If you’re using aggregation, set up LACP on both ends. 7) Test with iperf3 and real storage traffic to confirm throughput is meeting expectations.

If you’re new to NIC bonding or LACP, our tutorial on link aggregation in Linux offers a practical walkthrough and explains how to avoid common misconfigurations: {% post_url 2025-04-02-linux-lacp-tutorial %}.

### Troubleshooting common issues

- No link detected on one or both ports: ensure the fiber modules are correctly seated, the correct SFP+ module types are used, and the cables aren’t damaged. Check the switch port configuration to confirm it’s in a mode that supports SFP+ and that it isn’t limiting the speed to 1 Gbps.
- Uneven performance across ports: verify the modules’ correctness and MTU settings. Check if the switch has misconfigured QoS rules that cause one port to be deprioritized.
- Issues with bonding: confirm the switch supports the selected LACP mode and that both NICs are configured identically on both ends. Sometimes a firmware update on the switch or the NIC helps stabilize the bond.
